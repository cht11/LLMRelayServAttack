# 创建 GitHub 项目介绍推文

## 前置要求

**必须获取 GitHub 项目地址**：在开始之前，向用户索要 GitHub 项目 URL。

```
请提供您想要推荐的 GitHub 项目地址（例如：https://github.com/owner/repo）
```

如果用户没有提供有效的 GitHub 地址，不要继续后续步骤。

## 工作流程

### 步骤 1：询问是否 Clone 到本地

获取 GitHub 地址后，询问用户是否需要深度分析：

```
是否需要将项目 clone 到本地进行深度分析？

1. 是 - clone 到 materials/ 目录，分析 README、examples、docs、showcase 等
2. 否 - 仅通过在线 README 获取信息

深度分析可以获取更全面的项目信息，推荐用于复杂项目。
```

**注意**：`materials/` 目录不纳入 git 管理，仅作为分析素材使用。

### 步骤 2A：Clone 模式（深度分析）

如果用户选择 clone：

1. **Clone 项目到本地**：
```bash
mkdir -p materials
export http_proxy=http://127.0.0.1:7890 && export https_proxy=http://127.0.0.1:7890 && git clone --depth 1 <项目地址> materials/<项目名>
```

2. **深度分析项目内容**：
   - `README.md` - 项目介绍和基本用法
   - `docs/` - 详细文档
   - `examples/` - 使用示例代码
   - `showcase/` 或 `demo/` - 演示案例
   - 其他重要目录（如 `tutorials/`、`samples/`）

3. **提取关键信息**：
   - 项目名称和描述
   - 主要功能特性
   - 技术栈和依赖
   - 使用方法和示例
   - 项目亮点和应用场景

4. **复制图片到本地**：
   - 使用 `cp` 命令从 clone 的目录复制图片
   - 保存路径：`src/images/<项目名>-<图片序号>.<扩展名>`

```bash
mkdir -p src/images
cp materials/<项目名>/docs/images/screenshot.png src/images/<项目名>-1.png
```

### 步骤 2B：在线模式（快速分析）

如果用户选择不 clone：

1. **访问 GitHub 项目页面**：使用 WebFetch 工具访问用户提供的 GitHub URL
2. **读取 README 文件**：
   - 直接访问 raw README：`https://raw.githubusercontent.com/<owner>/<repo>/<branch>/README.md`
   - 常见分支名：`main` 或 `master`

3. **提取关键信息**：
   - 项目名称和描述
   - 主要功能特性
   - 技术栈
   - 使用方法
   - 项目亮点

4. **下载图片到本地**：
   - 使用 curl 下载（需配置代理）
   - 保存路径：`src/images/<项目名>-<图片序号>.<扩展名>`

```bash
mkdir -p src/images
curl -L --proxy http://127.0.0.1:7890 "https://github.com/owner/repo/raw/main/screenshot.png" -o "src/images/project-name-1.png"
```

### 步骤 3：处理图片

**重要**：README 中的所有图片必须下载/复制到本地。

1. **识别图片链接**：扫描 README 中的所有图片引用
   - Markdown 格式：`![alt](url)`
   - HTML 格式：`<img src="url">`

2. **保存图片到本地**：
   - Clone 模式：使用 `cp` 从 `materials/<项目名>/` 复制
   - 在线模式：使用 `curl --proxy http://127.0.0.1:7890` 下载

3. **记录图片映射**：建立原始路径到本地路径的映射，用于后续替换

### 步骤 4：与用户确认标题

根据项目信息，向用户建议文章标题：

```
根据该项目的特点，小编建议以下标题：
- 「开源推荐｜xxx：一句话描述项目亮点」
- 「GitHub 热门项目｜xxx 深度解析」
- 「安全工具推荐｜xxx 使用指南」

您觉得哪个标题更合适，或者有其他想法？
```

### 步骤 5：选择专栏分类

引导用户选择专栏（同 create-wechat-article）：

| 专栏 | 目录 | 适合内容 |
|------|------|----------|
| 逆向工程 | `reverse-engineering/` | 逆向分析、反汇编、调试工具 |
| 移动安全 | `mobile-security/` | iOS/Android 安全相关工具 |
| 训练营 | `training-camp/` | 学习资源、教程项目 |
| AI 与安全应用 | `ai-security/` | AI 安全、ML 相关项目 |

### 步骤 6：创建文章文件

**获取当前日期**：在创建文件前，先执行系统命令获取当前日期：

```bash
date +%Y-%m-%d
```

使用以下模板创建文章（将 `<当前日期>` 替换为上述命令输出）：

```markdown
---
创建日期: <当前日期>
作者 ID: xxx
审核 ID: xxx
是否发表: false
---

# [文章标题]

> 文章摘要：简要介绍这个开源项目及其核心价值

## 项目简介

[基于 README 内容，用小编的口吻介绍项目背景和用途]

## 核心功能

[列举项目的主要功能和特性，用连贯的文字讲解而非简单罗列]

## 项目亮点

[分析项目的技术亮点或独特之处]

## 总结

[对项目的整体评价和推荐理由]

## 相关文章推荐

- [相关文章标题](../专栏/文件名.md)

---

**项目地址**

https://github.com/owner/repo
```

**元信息字段说明**：
- **创建日期**：自动获取，格式 `年-月-日`
- **作者 ID**：可询问用户，或使用默认值 `xxx`
- **审核 ID**：可询问用户，或使用默认值 `xxx`
- **是否发表**：默认 `false`，发表后改为 `true`

**其他注意事项**：
- 文件名格式：`github-<项目名>.md`
- 图片引用使用相对路径：`../images/<图片文件名>`
- **结尾的项目地址必须是纯文本格式**，不要使用 Markdown 链接
- **不要写项目安装和环境配置**：文章聚焦于项目介绍和功能亮点，不需要详细的安装步骤或环境搭建指南

### 步骤 7：更新 SUMMARY.md

在 `src/SUMMARY.md` 对应专栏下添加文章链接，子文章缩进两个空格。

## 写作规范

### 人称使用

使用"小编"作为主语，而不是"我"或"我们"。

```markdown
❌ 错误示例：
我今天要给大家介绍一个有趣的项目...
我们来看看这个工具是怎么工作的...

✅ 正确示例：
小编今天要给大家介绍一个有趣的项目...
接下来小编带大家看看这个工具是怎么工作的...
```

### 讲解风格

采用由浅入深的教学方式，像老师授课一样用连贯的文字讲解，专注于第一原理（First Principles），而非知识点罗列。注意理解知识层级，不要对过于基础的知识进行过多解释。

```markdown
❌ 错误示例（列点式总结）：
该项目主要特点：
- 高性能
- 易扩展
- 跨平台

✅ 正确示例（连贯讲解）：
这个项目为什么值得关注？小编先带大家思考一个问题：当我们需要处理大量并发请求时，传统方案往往会遇到什么瓶颈？

答案是资源调度开销。而这个项目巧妙地采用了协程池设计，让性能得到了质的提升...
```

### 承上启下

在适当位置添加过渡性文字，帮助读者理解内容之间的逻辑关系。

常用过渡句式：
- "理解了这个概念之后，小编再带大家看看..."
- "说到这里，可能有读者会问..."
- "有了前面的基础，接下来的内容就好理解了..."
- "....."

这些只是过渡句式的示例，具体使用时根据内容灵活调整，不要机械套用。

### 避免机械连接词

不要使用"首先"、"其次"、"最后"、"第一"、"第二"等机械的序号式连接词。用更自然的过渡方式组织内容。

```markdown
❌ 错误示例：
首先是开源免费。其次是现代化界面。最后是活跃社区。

✅ 正确示例：
说到开源免费，这对个人学习者意义重大...
除了成本优势，Cutter 的界面设计也值得一提...
当然，一个好工具离不开社区的支持...
```
